This is the **Software Requirements Specification (SRS) for the SaleOS-Facebook Microservice**. Following your architecture, this service focuses on Meta Graph API integration, Facebook Page feed management, and the high-conversion "Comment-to-Messenger" pipeline.

# **Software Requirements Specification (SRS)**

## **Module: SaleOS-Facebook Microservice**

## **1\. Microservice Overview**

* **Microservice Name:** SaleOS-Facebook  
* **Architecture Role:** Asynchronous Page engagement, Messenger automation, and public feed management.  
* **Core Objective:** To manage Facebook Page presence, automate interactions under posts, and transition public commenters into private, AI-driven Messenger checkout funnels.  
* **Dynamic Capability:** Merchants link their Facebook Business Page via Meta OAuth. The service manages long-lived Page Access Tokens and handles event-driven webhooks for feed interactions and messages.

## **2\. Mandatory Tech Stack**

### **2.1 Frontend (Standalone UI/UX)**

* **Framework:** Vite \+ React \+ TypeScript (Strict Mode).  
* **Components:** Shadcn UI for structured data tables (Comment/Message logs); Drag-and-drop media uploaders.  
* **API Communication:** Axios (centralized instances) with JWT-based Auth.

### **2.2 Backend & Facebook Integration**

* **Framework:** FastAPI (Python 3.12+).  
* **Integration Library:** Official **Meta Graph API** (Messenger Platform, Feed API).  
* **Async Workflow:** httpx for API interactions; Celery/Redis for managing high-volume message queues and post-scheduling.

## **3\. Dynamic Configuration & Auth**

### **3.1 Meta OAuth Lifecycle**

* **Flow:** Merchants trigger an OAuth redirect via the UI, granting permissions for pages\_manage\_engagement, pages\_messaging, and pages\_read\_user\_content.  
* **Token Management:** The microservice exchanges short-lived codes for **60-day Long-Lived Page Access Tokens**. These are encrypted and stored in facebook\_bot\_configs. The service includes a background task to refresh these tokens automatically to prevent connection lapses.

## **4\. Database Requirements (Facebook Schema)**

#### **4.1 Table: facebook\_bot\_configs**

* **Purpose:** Stores Page-specific credentials and subscription states.  
* **Columns:**  
  * id (UUID, PRIMARY KEY)  
  * merchant\_id (UUID, FOREIGN KEY, UNIQUE)  
  * page\_id (VARCHAR(255), NOT NULL)  
  * access\_token (TEXT, Encrypted, NOT NULL)  
  * is\_connected (BOOLEAN, Default: FALSE)  
  * automation\_enabled (BOOLEAN, Default: TRUE)

#### **4.2 Table: facebook\_conversations**

* **Purpose:** Maps Messenger threads to SaleOS orders.  
* **Columns:**  
  * id (UUID, PRIMARY KEY)  
  * merchant\_id (UUID, FOREIGN KEY)  
  * psid (VARCHAR(255), NOT NULL) — *Page Scoped ID for the customer.*  
  * last\_sync\_at (TIMESTAMP)

## **5\. API Specification**

### **5.1 Internal Dashboard APIs**

* **POST /api/v1/facebook/auth/connect**: Initiates Meta OAuth flow.  
* **POST /api/v1/facebook/broadcast**: Publishes media/text to the Page feed.  
* **GET /api/v1/facebook/inbox**: Fetches active Messenger threads for the UI.

### **5.2 External Meta Webhook**

* **POST /api/v1/facebook/webhook/{merchant\_id}**: Listens for feed and messages events.  
  * *Logic:*  
    1. Identifies if the event is a comment or a message.  
    2. Routes to the **Central AI Core** for intent analysis.  
    3. Executes the Private Reply via the Graph API’s /messages endpoint.

## **6\. UI/UX Requirements (React Dashboard)**

* **Page Management Panel:** View connected Facebook Pages, toggle auto-replies on/off, and see status indicators for API connectivity.  
* **Comment Feed:** A real-time table showing public comments. Each row includes an "Escalate to Messenger" button (if not already triggered) and a status tag: \[Replied\], \[Messenger Chat Open\], \[Order Created\].  
* **Media Broadcast Studio:** An interface to upload content that simultaneously posts to Facebook, Instagram, and TikTok, using the metadata generated by the AI Hub.  
* **Validation:** Custom Zod schemas to ensure all Facebook image/video uploads adhere to Meta's strict aspect ratio and resolution guidelines.

## **7\. Operational Workflow (The "Messenger Conversion" Loop)**

1. **Event Ingestion:** Customer comments on a product post. Meta pushes the webhook.  
2. **Contextual Brain:** The microservice pulls the comment, queries the **Central AI Core**, and verifies the product in the master inventory\_ledger.  
3. **Public/Private Engagement:** The backend replies publicly to the comment, then uses the comment\_id to initiate a private Messenger DM.  
4. **Transaction Flow:** The customer interacts with the AI agent in Messenger. Once the product is selected, the customer sends a payment receipt (Telebirr/CBE).  
5. **OCR Verification:** The microservice routes the image to the **External Ethiopian Bank OCR API**.  
   * **Success:** Order status PAID is pushed to the master database.  
   * **Manual Fallback:** If the OCR fails, the microservice alerts the merchant's Facebook dashboard and instructs the AI agent to provide human support contact details.

**Architectural Note:** Meta's **24-hour messaging window** is critical. The microservice logic is hard-coded to respect this: if the window has closed, the AI agent is programmed to send a "message tag" request or invite the user to re-engage with the bot, preventing API 400 errors for out-of-window message attempts.
